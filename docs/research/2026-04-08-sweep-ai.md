# Sweep AI — AST-Aware Autonomous Issue Fixing (2024)

## Overview

Sweep is an AI-powered GitHub bot that autonomously converts GitHub issues into pull requests. Key insight: the hardest part of autonomous code fixing is not generation — it's **deciding which code is relevant** and **ensuring multi-file changes stay consistent**. Sweep addresses both with AST bipartite graphs and topological diff propagation.

**Typical latency**: 3-5 minutes issue-to-PR comment.

## Core Pipeline

1. **Issue Ingestion** → tag issue with "sweep" label
2. **Context Retrieval** → hybrid TF-IDF + embedding vector search over file chunks
3. **AST Bipartite Graph** → files ↔ code entities (functions/classes); prune zero-degree edges
4. **Planning** → issue → file-level instructions using entity-level chunking (~150-200 lines/entity)
5. **Code Generation** → entity-level context window (~10-15k tokens); reduces context noise
6. **Diff Propagation** → topological sort ensures signature changes cascade across files
7. **Validation Loop** → run tests → detect failures → auto-fix → re-run
8. **PR Submission** → commit + test results flowchart

## Key Architectural Patterns

### 1. Hybrid Retrieval (Avoids RAG "Lost-in-the-Middle")

Combines TF-IDF lexical search + embedding-based vector similarity. Pure RAG fails because relevant files are buried in long context. Hybrid approach retrieves candidates from both signals and intersects.

**For Clade**: TLDR injection is one-pass. A hybrid approach (keywords + structure) would better identify relevant context for complex bug fixes.

### 2. AST Bipartite Graph + Entity Chunking

Builds a bipartite graph: files ↔ code entities (functions/classes). Prunes edges based on relevance to the issue. Result: 680 edges → 102 edges in one example. The agent only sees relevant entities, not whole files.

**For Clade**: `_parse_python_ast` already extracts function signatures. But the worker gets everything, not a pruned entity-level view. Adding pruning (keep only entities mentioned in the issue or directly called by them) would reduce context noise significantly.

### 3. Topological Diff Propagation

When function signatures change, all callers must be updated. Sweep builds a call graph, sorts topologically, and generates diffs in order. Prevents inconsistent multi-file edits where File A has the new signature but File B still passes old args.

**For Clade**: Workers sometimes change function signatures without updating callers (then fail oracle). Generating a call-dependency hint list as part of the task context would help.

### 4. Test-Driven Validation Loop

Sweep runs the project's test suite against the generated patch. If tests fail, it autonomously:
1. Identifies which test failed and why
2. Generates a targeted fix
3. Re-runs tests
4. Repeats up to N times

**For Clade**: `_run_lint_check()` handles syntax/style errors. But no functional test runner. Adding `pytest --tb=short` (or project's test command) to the post-worker verification would catch functional regressions.

## Gaps vs Clade's Current Implementation

### §Gap 1 — Entity-Level Context Pruning

Worker gets entire TLDR blob. Sweep prunes to issue-relevant entities only.

**Fix**: Extend `_localize_fault()` to also return a list of entity names (function/class names). In `_build_task_file()`, filter the TLDR to only show sections containing those entities. Reduces context noise 3-5×.

### §Gap 2 — Call-Site Dependency Hints

When changing a function signature, the worker needs to know which callers exist. No call graph info today.

**Fix** (small): After TLDR localization, add a "callers" hint block: `grep -rn "def <function_name>" → extract callers`. Simple Bash, no index required. Tell the agent: "If you change these functions, also update these callers."

### §Gap 3 — Post-Worker Test Runner

No functional test execution after worker completes. Only lint check.

**Fix**: Add optional `test_cmd` to `.claude/orchestrator.json`. After `verify_and_commit()` succeeds, run `test_cmd` with a 60-second timeout. If tests fail, inject failures into reflection retry. Small effort, high value for projects with test suites.

### §Gap 4 — Hybrid Context Retrieval

TLDR is pure structure-based. For bug fixes, the bug description keywords should also guide which files are retrieved.

**Fix**: Enhance `_localize_tldr_for_task` to combine keyword matches (grep the TLDR for issue terms) with the haiku structural selection. Two-signal retrieval for better precision.

## Key Actionable Items

1. **§Gap 3 (post-worker test runner)** — Small effort. Add `test_cmd` support in `orchestrator.json` + run after commit. Directly catches functional regressions the lint check misses.

2. **§Gap 2 (caller hints)** — Small effort. After fault localization identifies suspect functions, grep for callers and inject as a "If you change X, also update these callers" hint. Prevents cascading signature breakage.

3. **§Gap 1 (entity-level TLDR pruning)** — Medium effort. Filter TLDR sections to only show entities relevant to the task. Reduces context noise significantly.

4. **§Gap 4 (hybrid retrieval)** — Medium effort. Combine keyword grep with structural haiku selection in `_localize_tldr_for_task`.
