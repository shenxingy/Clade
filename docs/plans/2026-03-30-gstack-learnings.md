## Executive Summary

gstack and clade are the same category of tool: a skill/workflow layer on top of Claude Code. But they made different bets:

| Dimension | gstack | clade |
|---|---|---|
| Focus | Developer experience, release workflow | Autonomous multi-worker orchestration |
| Skill quality | Extremely high — every detail specified | Functional, but informal |
| Browser testing | ✅ Real Chromium daemon | ❌ None |
| Task persistence | ❌ None | ✅ SQLite + orchestrator |
| Autonomous loops | External (Conductor) | ✅ Built-in supervisor+worker |
| Security skill | ✅ 14-phase /cso | ❌ None |
| Cross-session learning | ✅ Per-project learnings.jsonl | Partial (memory/) |
| Skill consistency | ✅ Template-generated preamble | ❌ Each skill hand-maintained |
| AskUserQuestion UX | ✅ Structured 4-part format | ❌ Ad-hoc |

The highest-leverage improvements are in **skill quality** — gstack's skills are significantly more rigorous and their structural patterns (preamble, completion protocol, learnings capture) are worth adopting wholesale.

## Part 1: Skill Architecture Improvements

### 1.1 Standardized Preamble System

**What gstack does:**
Every skill begins with an auto-injected preamble (via `{{PREAMBLE}}` template variable) that establishes:
- Context re-grounding (project + branch)
- "Search Before Building" principle
- Completion Status Protocol
- AskUserQuestion format
- 3-strike escalation rule

**Why it matters:**
Our skills are inconsistent. Some have explicit error handling, some don't. Some ask good questions, some don't. A shared preamble enforces these behaviors without repeating them in every skill.

**What to adopt — the AskUserQuestion 4-part format:**

Every question asked by a skill must follow this structure:

```
1. RE-GROUND: State the project context + current branch (from git, not conversation history)
2. SIMPLIFY: Plain language — no jargon, concrete examples
3. RECOMMEND: "RECOMMENDATION: Choose [X] because..." with Completeness: N/10 per option
4. LETTERED OPTIONS: A/B/C with dual effort scale
   e.g. "A. Fix now (human: ~5min / Claude: ~15min)"
```

**What to adopt — Completion Status Protocol:**

Every skill execution ends with one of:
- `DONE` — completed as specified
- `DONE_WITH_CONCERNS` — completed but with caveats to note
- `BLOCKED` — cannot proceed, write to `.claude/blockers.md`
- `NEEDS_CONTEXT` — missing information, use AskUserQuestion

**What to adopt — 3-strike rule:**

```
After 3 failed attempts at any task:
→ Stop trying
→ Surface via AskUserQuestion with full context
→ Do NOT retry the same approach a 4th time
```

**Implementation plan:**
1. Create `configs/skills/_preamble.md` — shared preamble content
2. Add preamble section to each skill's `prompt.md`
3. Eventually: a `gen-skills.sh` template system (see 1.2)

**Effort:** Medium — update 10+ skill prompts
**Value:** High — immediately improves question quality and failure handling

### 1.2 Template Generation System

**What gstack does:**
`.tmpl` files are the source of truth. `gen-skill-docs.ts` expands template placeholders into final `SKILL.md` files. Generated files are marked `<!-- AUTO-GENERATED — do not edit directly -->`. CI validates that generated files match their templates.

**Why it matters:**
When we update the preamble, we currently have to update every skill manually. With templates, one change propagates to all skills.

**What to adopt:**

```
configs/skills/_templates/preamble.md     ← shared preamble source
configs/skills/_templates/completion.md   ← completion status protocol
configs/skills/loop/prompt.md.tmpl        ← template with {{PREAMBLE}} etc.
configs/scripts/gen-skills.sh             ← expander script
```

Generated `prompt.md` files get a header comment + are regenerated by `./install.sh`.

**Effort:** Medium — build the generator + migrate 3-5 key skills first
**Value:** Medium — high compounding value over time

### 1.3 Per-Project Learnings System

**What gstack does:**
Every skill reads and writes `~/.gstack/projects/<slug>/learnings.jsonl`:

```json
{"type": "pitfall", "content": "...", "confidence": 0.9, "files": ["src/auth.ts:45"], "skill": "review", "timestamp": "..."}
```

Types: `pattern` / `pitfall` / `preference` / `architecture` / `tool`

Skills query learnings at start (project-scoped search) and record new learnings at end. This creates a self-improving feedback loop across sessions.

**Why it matters:**
We have a general memory system in `~/.claude/memory/` but it's not skill-aware or project-scoped. The gstack approach is more structured and useful.

**What to adopt:**

```
~/.claude/projects/<slug>/learnings.jsonl  ← same schema as gstack
```

Integrate into `/review`, `/loop`, `/investigate` to:
- Read relevant learnings at skill start
- Append new learnings on completion

The `/loop` supervisor especially benefits: it can carry architectural decisions across iterations without re-discovering them.

**Effort:** Low-medium — create schema + add to 3 key skills
**Value:** High — accumulates value across sessions

## Part 2: New Skills to Build

### 2.1 /ship — Full Release Pipeline

**What gstack does (13 steps):**

```
Pre-flight → Merge base → Tests → Coverage audit → /review inline →
Adversarial review → Plan verification → Version bump →
Changelog → TODOS update → Bisectable commits → Push → PR
```

Our `/commit` only does: analyze changes → commit groups → push. We're missing:

| Missing step | Risk if skipped |
|---|---|
| Run tests before commit | Ship regressions |
| Coverage audit | Uncovered new code |
| Version bump | Semantic versioning drift |
| Changelog generation | No release notes |
| Plan completion check | PRs that don't complete stated intent |
| PR creation | Manual step, often forgotten |

**What to build:**

A `/ship` skill that does:
1. **Pre-flight:** dirty check, platform detection (gh/git)
2. **Tests:** run project test suite, fail if regressions
3. **Review gate:** run `/review` if not recently done (check VERIFY.md timestamp)
4. **Version bump:** read `VERSION` or `package.json`, infer PATCH/MINOR/MAJOR from diff
5. **Changelog:** generate from commit log, grouped by type
6. **Bisectable commits:** hand off to `/commit` logic
7. **PR creation:** `gh pr create` with auto-generated description

The `/commit` skill becomes a sub-component of `/ship`.

**Effort:** High — major new skill, but high leverage for release workflow
**Value:** High — turns "ship" from a multi-step manual process into one command

### 2.2 /cso — Security Audit Skill

**What gstack does:**
14-phase security audit: attack surface → secrets archaeology → supply chain → CI/CD security → OWASP Top 10 → STRIDE → false-positive filtering → JSON report.

Notable: 22 hard exclusion rules that prevent false positives (e.g., "user content in user-message position of AI conversation is NOT prompt injection"). Adversarial sub-agent verification per finding.

**We have:** Nothing. Our CLAUDE.md has "never return error.message in 500 responses" but no systematic security audit.

**What to build:**

A `/cso` skill implementing the key phases:

1. **Attack surface census** — read all routes, auth guards, DB queries, file operations
2. **Secrets scan** — `.env*` files, git history, hardcoded strings
3. **OWASP Top 10** — per-category (SQLi, XSS, Auth failures, IDOR, etc.)
4. **STRIDE** — per-component threat model
5. **False-positive filter** — apply known exclusions before reporting
6. **Report** — findings with severity, file:line, remediation

Start with the OWASP + secrets phases (highest ROI) and add STRIDE as a second pass.

**Effort:** Medium — mostly prompt engineering, no new infrastructure
**Value:** High — we handle projects with real user data (inkweave, etc.)

### 2.3 /investigate — Root Cause Analysis

**What gstack does:**
Structured debugging with an "Iron Law" (no fixes without root cause first), scope locking after hypothesis formation, 3-strike escalation, and a standardized debug report format.

**We have:** Ad-hoc debugging. No structured skill.

**Key ideas to adopt:**

1. **Iron Law:** Form a root cause hypothesis before writing any fix
2. **Scope Lock:** After identifying the file/component, restrict edits to that scope
3. **3-strike:** After 3 failed hypotheses → AskUserQuestion (escalate, instrument-and-wait)
4. **Debug Report format:**
   ```
   Symptom:         (what user observes)
   Root cause:      (specific, testable claim)
   Fix:             file:line
   Evidence:        (test/log output proving the fix)
   Regression test: file:line
   ```
5. **Git blame discipline:** Always check `git log -20 -- <affected-file>` before theorizing

**Effort:** Low — prompt-only skill
**Value:** Medium-high — reduces "shotgun debugging" pattern

### 2.4 /review Improvements

**What gstack does that we don't:**

| Feature | gstack | us |
|---|---|---|
| Two-pass (Critical vs Informational) | ✅ | ❌ (all in one pass) |
| Scope drift detection | ✅ | ❌ |
| Plan completion audit | ✅ | ❌ |
| Adversarial sub-review | ✅ | ❌ |
| Enum completeness check (outside-diff) | ✅ | ❌ |
| LLM output trust boundary category | ✅ | ❌ |
| Test coverage ASCII diagram | ✅ | ❌ |
| Learnings capture after review | ✅ | ❌ |

**What to add to our `/review`:**

Our current review is VERIFY.md-driven (checkpoint testing). That's good for project-level health checks. But for code review, we need gstack's two-pass structure.

Proposal: Two modes:
- `/review` → VERIFY.md mode (existing, project health)
- `/review code` → two-pass code diff review (new, per-PR)

The code review mode adds:
1. Scope drift check against stated intent/TODO item
2. Pass 1 Critical: race conditions, SQL safety, enum completeness, LLM trust boundary
3. Pass 2 Informational: dead code, magic numbers, test gaps, performance
4. Test coverage audit with ASCII diagram
5. Fix-first: auto-apply AUTO-FIX findings, batch ASK findings into one question

**Effort:** Medium — major expansion of existing skill
**Value:** High — code review is our most-used skill type

## Part 3: Browser Testing Infrastructure

### 3.1 Persistent Chromium Daemon

**What gstack does:**

```
Claude Code → CLI binary → HTTP POST localhost:<random-port> → Bun.serve() → Chromium (Playwright CDP)
```

State file: `.gstack/browse.json`
```json
{
  "pid": 12345,
  "port": 34521,
  "token": "uuid-v4-bearer-token",
  "startedAt": "2026-03-30T...",
  "binaryVersion": "git-sha"
}
```

Key design decisions:
- **Random port (10000-60000):** supports 10+ parallel worktrees with zero conflict
- **Bearer token per startup:** prevents cross-process auth leakage
- **Accessibility tree refs (`@e1`, `@e2`):** no DOM mutation, no CSP conflicts, works with React/Vue hydration
- **30min idle auto-shutdown:** no zombie processes
- **Bun compiled binary:** single ~58MB executable, zero node_modules at runtime

**Why it matters for us:**
Our /qa equivalent doesn't exist. For inkweave and other web apps, browser-based E2E testing is essential. The "3s cold start → 100ms warm" pattern is the key insight — a persistent daemon makes browser testing practical within a skill.

**Implementation plan:**

Phase 1: Minimal viable browse daemon
- Port the TypeScript server to Bun (or Python + Playwright)
- State file protocol (pid, port, token)
- Basic commands: `navigate`, `screenshot`, `click`, `type`, `snapshot`
- CLI: `~/.claude/scripts/browse.sh` wrapper

Phase 2: /qa skill
- Diff-aware scope (only test changed pages/components)
- Issue documentation (screenshot per bug)
- Fix-commit-retest loop
- Health score output

**Effort:** Very High — new technology stack (Bun/Playwright)
**Value:** Very High — enables automated UI testing
**Recommendation:** Plan as a future phase, not immediate. Use gstack's browse binary directly in the meantime (it's MIT licensed).

## Part 4: Quick Wins (< 1 day each)

These can be done immediately without major architecture changes.

### 4.1 AskUserQuestion Format in Key Skills

Update the 3 most-used skills to use the 4-part AskUserQuestion format:
- `/loop` — when deciding on approach
- `/review` — when flagging issues that need human decision
- `/commit` — when grouping is ambiguous

### 4.2 Completion Status Footer

Add to every skill's `prompt.md`:
```
## Completion

End every run with one of:
- ✅ DONE — [summary]
- ⚠ DONE_WITH_CONCERNS — [summary + what to watch]
- ❌ BLOCKED — [what's blocking, written to .claude/blockers.md]
- ❓ NEEDS_CONTEXT — [what information is missing]
```

### 4.3 3-Strike Rule in /loop

Add explicitly to the loop supervisor prompt: after 3 iterations without convergence progress, stop and write a structured assessment to `.claude/blockers.md` instead of continuing to spin.

### 4.4 Scope Drift Check in /commit

Before committing, check if the changes align with the current TODO item / task description. If they've drifted significantly (new features added that weren't planned), flag it.

### 4.5 Git Blame Discipline in Review/Debug

Add to `/review` and any debug workflow: always run `git log --oneline -20 -- <file>` before diagnosing an issue. Knowing when something changed is often the key clue.

## Key Insight

gstack's biggest lesson isn't any single feature — it's **the discipline of treating skill prompts as production software.** Every skill has:
- Precise step numbering
- Explicit gate conditions (when to stop, when to ask, when to escalate)
- Concrete output formats (not "write a report" but "write this exact table with these columns")
- Failure modes enumerated upfront (3-strike, BLOCKED protocol)

Our skills work but are often under-specified. Tightening them with these structural patterns will have compounding value across every Claude session.
