## 2026-03-04 — GUI 平台愿景（从 2026-03-02 brainstorm 恢复，之前 refine 时被错误简化）

> 来源：2026-03-02 session `9840664c` 中用户的完整描述 + 2026-03-04 对话中的补充。
> 之前 refine 到 VISION.md 时只保留了"monitoring cockpit"定义，丢掉了下面这些核心交互设计。

### 核心定位：不是监控面板，是 AI SDE 操作台

部署在网上，任何设备（手机/电脑）都能访问。不只是看状态，而是用户与 AI 协作的主入口。

### A. 想法收集（低摩擦、异步、不阻塞）

用户的原话：
> "用户输入的时候，一直自己一个人说...可能我在读AI生成结果时，已经把想到的东西忘掉了...用户输入可以一直说，然后在旁边显示建议"
> "能够一口气说很多的问题，而不是说一个问题AI改，说一个问题AI改"
> "不应该有很大阻力，也许他一边玩的时候，就可以一边把这个问题收集起来"

**设计要点：**
- 用户可以一条一条往下输入想法，不需要等 AI 回复
- AI 在旁边（侧栏？实时批注？）异步显示评估和建议
- 输入不阻塞 — 不是 TUI 那种"说一句等一句"的串行模式
- 想法自动归集，不丢失
- 手机上也能用（最低摩擦的想法入口）

### B. AI 协商 → 拍板执行

用户的原话：
> "点子不一定是好的，说不定还有更好的方案...需要跟AI一起把它完善，然后觉得OK了，那就正式拍板去做"
> "AI做引导...不是被动等指令，而是主动问"

**设计要点：**
- AI 评估每条想法：可行性、风险、是否有更好方案
- AI 主动引导：提问未考虑的场景、挑战假设
- 双方都同意 → 想法升级为正式目标（写入 VISION/TODO）
- 正式目标 → 后台 Claude Code 自动开始实现
- 不同意的想法保留在收集区，不删除

### C. 跨项目调度 Dashboard

用户的原话：
> "整个系统平台能像我现在需要优化几个项目一样，横跳操作完全统筹"
> "能够在不同设备上直接操作，控制不同的project，看看它们跑了多少"

**设计要点：**
- 所有项目一览：状态、进度、花费、blocker
- 启停 start.sh 长时间运行
- 查看 agent 状态：正在跑、完成了、中间碰到问题
- start.sh 在后台作为执行引擎，GUI 是控制面板

### D. 后台自主维护

用户的原话：
> "定期去跑一跑，review一下，看看有没有什么东西可以更改的"
> "用户提供反馈的地方，如果用户提供反馈了，他可能也会自己去评估一下"

**设计要点：**
- 系统定期自动巡查各项目：跑 review、检测 bug、检查 CI
- 发现问题 → 自己修（绿区）或提报给用户（黄/红区）
- 巡查中产生的新想法 → 回到收集区
- 用户反馈入口：用户提交反馈 → AI 评估是否合理 → 融合到项目中

### E. 与现有系统的关系

- `start.sh` = 后台执行引擎（已有，继续用）
- `orchestrator/` = 需要从当前的"开发者调试面板"升级为上述操作台
- CLI 层 = 底层原语不变，GUI 包装 CLI
- BRAINSTORM.md / TODO.md / VISION.md = 数据层，GUI 读写这些文件

### 未解决的设计问题

1. **异步交互怎么实现？** — 用户持续输入 + AI 旁路反馈，技术方案未定（WebSocket streaming？双栏 UI？）
2. **AI 引导不应该有延迟** — 但 AI 需要时间想。矛盾怎么解决？（预加载？流式输出？）
3. **前端审美问题** — AI 生成的 UI 质量不够，Phase 12.3 design system 是一个方向，但够不够？
4. **部署方案** — Caddy HTTPS？Tailscale？公网？安全性？

## 2026-03-03 — 老照片整理 + AI 回忆录

**来源:** 和外公聊天 — 老年人想整理老照片，但手动归纳太费劲

**核心概念:** 扫描/导入照片 → AI 自动整理 → 可选生成回忆录
1. **照片整理层**: 扫描纸质照片 / 导入手机相册 → AI 识别人脸、场景、时间 → 自动分类（按人物、时间线、事件）→ 去重、修复（老照片增强）
2. **回忆录层** (与 Granola 项目交叉): 照片 + 口述录音 → AI 整理成回忆录章节（时间线 + 人物 + 故事）

**用户画像:** 老年人（操作必须极简） + 子女辅助（帮爸妈扫描、设置）

**技术拆解:**
- 照片扫描: 手机摄像头 + 自动裁切矫正（OpenCV / 现成SDK）
- 人脸聚类: face_recognition / InsightFace → 自动按人物分组
- 场景/时间识别: CLIP + EXIF + OCR（照片背面手写日期）
- 老照片修复/上色: 现有模型（CodeFormer, DDColor）
- 回忆录生成: 照片元数据 + 口述转写 → LLM 整理 (复用 Granola 的模板引擎)

**与 Granola 项目的关系:**
- 回忆录 = Granola 的一个模板（输入: 照片+口述 → 输出: 回忆录结构）
- 可以是 Granola 的子功能，也可以是独立 app
- 共享: ASR引擎、LLM整理引擎、模板系统

**风险/待验证:**
- 市场上可能有现成 app（Google Photos 已经做了很多自动整理）
- 老年人 UX 极难做（字要大、步骤要少、容错要高）
- 纸质照片扫描质量参差不齐
- 需要验证: 老年人愿意口述吗？子女愿意帮忙设置吗？

## 2026-03-03 — deepfake-platform 作为 kit 测试项目

**背景:** 工作项目，AI 相关（deepfake），有队友协作开发

**对 claude-code-kit 的测试价值:**
- AI/ML 项目类型（大文件、模型权重、GPU 依赖）→ 测试 kit 对 ML 项目的适配
- 真实漏洞存在 → 测试 security scan / code health 类 task factory
- 多人协作 → 测试 kit 在有队友的 repo 上的安全性（不能乱 push、要开 PR）

**⚠️ 约束（必须遵守）:**
- 工作项目 — 不暴露代码/业务细节到其他项目
- 有队友 — 所有改动必须走 PR，不能直接 push main
- 大文件 — .gitignore 敏感，模型权重/数据集不能误 commit
- **只做只读分析（audit/review）或开 PR 的改动，不做 force push / 直接 commit main**

**可以安全做的事:**
- `start.sh --goal` 跑 mypy/lint/test 类任务（只读分析 + PR）
- security audit（找漏洞、报告到 PR）
- code review（自动化 review 已有 PR）
- 但每次都要 `--dry-run` 先看计划
